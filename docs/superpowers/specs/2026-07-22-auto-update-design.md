# ars 자동 업데이트 설계

- 날짜: 2026-07-22
- 상태: 승인됨

## 목표

`ars`를 인터랙티브 모드로 실행할 때 새 릴리스가 있으면 TUI 진입 전에 안내하고,
Enter 한 번으로 업데이트한 뒤 새 버전으로 이어서 실행한다.

## 범위

- 대상: 인터랙티브 모드(`ars`, `ars <host>`)만. `ars list --json`,
  `ars remote add` 등 비인터랙티브 경로에서는 체크도 프롬프트도 하지 않는다.
- 설치 채널: npm 글로벌 설치(`@baleen37/ars`)와 GitHub Releases 단독
  바이너리 모두 Enter 한 번으로 업데이트를 완결한다.
- 소스 빌드(개발 빌드)는 업데이트 체크를 완전히 건너뛴다.

## 설계

### 1. 버전 임베드

- `cmd/ars-build`의 릴리스 빌드(`buildRelease`)가 ars 바이너리를 빌드할 때
  `-ldflags`에 `-X main.version=<semver>`를 추가한다.
- `cmd/ars`에 `var version string`을 두고, 빈 값이면 개발 빌드로 간주해
  업데이트 체크를 건너뛴다.

### 2. 버전 체크 (매 실행, 캐시 없음)

- 인터랙티브 모드 진입 직전에만 수행한다.
- `https://api.github.com/repos/baleen37/agent-remote-sessions/releases/latest`
  를 1.5초 타임아웃으로 조회해 `tag_name`(`v<semver>`)을 얻는다.
- 캐시를 두지 않는다. 매 실행 최대 1.5초 지연을 허용한다(사용자 결정).
- 네트워크 실패, 타임아웃, rate limit(비인증 60회/시간), 파싱 실패는 모두
  조용히 무시하고 TUI로 진입한다. 어떤 경우에도 시작을 막지 않는다.
- 최신 버전이 현재 버전과 같거나 비교 불가하면 프롬프트 없이 진입한다.

### 3. 프롬프트 (TUI 진입 전)

```
ars v1.3.0 사용 가능 (현재 v1.2.0)
Enter: 지금 업데이트    다른 키: 건너뛰기
```

- Enter만 업데이트를 실행한다. 그 외 모든 키(Esc, q 포함)는 건너뛰고 바로
  TUI에 진입한다.

### 4. 업데이트 적용 — 설치 채널 자동 감지

실행 파일 경로(`os.Executable`)로 채널을 판별한다.

- **npm 설치**: 경로에 `node_modules/@baleen37/ars`가 포함되면
  `npm install -g @baleen37/ars@<버전>`을 실행한다. npm 출력은 그대로
  사용자에게 보여준다.
- **단독 바이너리**: 해당 릴리스의 `ars_<ver>_<os>_<arch>.tar.gz`와
  `SHA256SUMS`를 다운로드하고, 체크섬을 검증한 뒤 아카이브에서 `ars`를
  추출해 임시 파일에 쓰고 현재 실행 파일 위치로 원자적으로 rename 한다.
  README의 수동 설치 절차와 동일한 검증 수준이다.

### 5. 업데이트 후

- 성공하면 새 바이너리를 같은 인자로 `exec` 해서 사용자가 끊김 없이 새 버전
  TUI로 진입한다. (npm 채널은 npm 런처 경로를 다시 실행한다.)
- 실패하면(권한 오류, 네트워크 오류 등) 에러 한 줄을 출력하고 현재 버전으로
  계속 진행한다. 어떤 실패도 ars를 사용 불가 상태로 만들지 않는다.

## 테스트

- 버전 비교, 채널 감지, 프롬프트 키 처리: 기존 스타일대로 의존성 주입 기반
  단위 테스트.
- 릴리스 조회, 체크섬 검증, tar 추출, 원자적 교체: `httptest` 가짜 서버와
  임시 디렉터리로 테스트.
- ldflags 버전 임베드: `ars-build` 릴리스 테스트에 검증을 추가.

## 결정 기록

- 캐시 없음: 24시간 캐시 제안을 사용자가 거절("더 짧게"). 매 실행 체크로
  단순화.
- 프롬프트 위치: TUI 시작 전 단독 프롬프트. TUI 안에서 Enter는 이미 세션
  attach에 사용 중이다.
- 완전 자동 교체(무확인)는 하지 않는다. Enter 확인 한 번을 거친다.
